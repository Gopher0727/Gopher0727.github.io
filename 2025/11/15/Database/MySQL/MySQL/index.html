基础

一、执行一条 SQL 查询语句的全流程

MySQL 架构分为 Server 层和存储引擎层。Server 层负责建立连接、分析和执行 SQL，存储引擎层负责数据存储和提取。

  1. 连接 MySQL 服务，需要 TCP 三次握手；建立连接后，连接器开始验证用户名和密码，获取用户权限。
    （建立的连接修改权限不会影响已经存在的连接权限）

  • show processlist 查看服务连接数

  • wait_timeout 参数控制空闲连接的最大空闲时长，默认是 8 个小时，可以通过 kill connection +id 断开连接（断开的是空闲连接还是活动连接需要看具体权限与配置）

  • max_connections 参数控制服务最大连接数

  • 怎么解决长连接内存占用？

    定期断开长连接
    客户端主动重置：v5.7 有 mysql_reset_connection 接口函数，将连接恢复到初始创建完的状态，不需要重连和权限验证

  1. 查询缓存（v8.0 移除了 Server 层的查询缓存）

  2. 解析 SQL（词法分析、语法分析）
    （解析器不会去查表或者字段存不存在）

  3. 执行 SQL

  • prepare
  • optimize

    在查询语句前面加上 explain 命令，看 SQL 语句的执行计划
    查询优化器发现某个值出现占比高于百分比界限（惯用 30%）时，会忽略索引，进行全表扫描

  • execute（与执行器的交互是以记录为单位）

    主键索引查询
    全表扫描
    索引下推：减少二级索引在查询时的回表操作

二、一行记录是怎么存储的？

进入 database 对应目录中，会有 db.opt / t_order.frm / t_order.ibd 三个文件，分别存储数据库默认字符集和字符校验规则、表结构及表的元数据信息、表数据（v5.6.6 之后，每张表数据都是一个独立的 .ibd 文件）

表空间由段、区、页、行组成的。

记录是按照行来存储，InnoDB 的数据是按页为单位读写（默认每个页 16 KB），数据表中的行记录就是用数据页来管理。

默认数据页包含文件头（38 字节）、页头（56 字节）、最大/最小记录（26 字节）、用户记录、空闲空间、页目录、文件尾（8 字节）七部分组成。
文件头包含前后两个指针，形成双向链表。

InnoDB 存储引擎是用 B+ 树来组织数据，B+ 树中每一层都是通过双向链表连接。在表中数据量大的时候，为某个索引分配空间就不按照页为单位，而是按区为单位（1 MB，相当于连续的 64 个页）进行分配，这样链表中相邻的页物理位置页相邻，就能使用顺序 IO 了。

段一般分为数据段（叶子节点）、索引段（非叶子节点）、回滚段（回滚数据）

行格式就是一条记录的存储结构。InnoDB 提供 Redundant / Compact / Dynamic / Compressed 四种行格式。

Compact 行格式

记录的额外信息（变长字段长度列表、NULL 值列表、记录头信息）

  1. 变长字段的真实数据占用的字节数按照列顺序逆序存放
    （记录头信息中的下一个记录的指针，指向下一条记录记录头信息和真实数据之间的位置，可以让位置靠前的记录的真实数据和数据对应的字段长度信息在同一个 CPU Cache Line，提高 Cache 命中率）
  2. NULL 值不会存放在行格式中记录的真实数据部分里。NULL 值列表也是逆序存储，用每一位表示每个数据是否为 NULL
    （所以数据库字段通常定义为 NOT NULL，至少减少一个字节的空间）
  3. 记录头信息（delete_mask / next_record / record_type …）

记录的真实数据（row_id / trx_id / roll_ptr …）

  1. row_id: （6 个字节）如果建表时没有指定主键或唯一约束，InnoDB 就为记录添加 row_id 隐藏字段
  2. trx_id: （6 个字节）事务 ID
  3. roll_ptr: （7 个字节）记录上一个版本的指针

MySQL 规定除了 TEXT、BLOBS 这种大对象之外，其他所有的列（不包括隐藏列和记录头信息）占用的字节长度加起来不超过 65535 字节。在计算 varchar(n) 中 n 的最大值时，需要减去变长字段长度列表、NULL 值列表所占用的字节数。

行溢出：一般数据放到数据页，溢出的数据放到溢出页。
发生行溢出时，记录的真实数据保存该列的一部分数据，剩余数据保存到溢出页，真实数据处用 20 字节指向溢出页地址。

索引

索引就是数据的目录，帮助存储引擎快速获取数据

索引分类

按数据结构分类：B+ 树索引、Hash 索引、全文（Full-text）索引
按物理存储分类：聚簇索引（主键索引）、二级索引（辅助索引）
按字段特性分类：主键索引（PRIMARY）、唯一索引（UNIQUE）、普通索引、前缀索引
按字段个数分类：单列索引、联合（复合）索引

  • InnoDB 不支持 Hash 索引，但是内存结构由自适应哈希索引
  • 如果有主键，InnoDB 使用主键作为聚簇索引的 key，否则就选择第一个不包含 NULL 值的唯一列作为 key，如果也没有就自动生成一个隐式自增的 id 列作为 key
  • 主键索引和二级索引 B+ 树的区别：主键索引 B+ 树叶子节点存放实际数据，二级索引 B+ 树叶子节点存放主键值，再通过主键索引查询到对应的叶子节点信息（回表），在二级索引的 B+ 树直接查询到结果的过程叫做 覆盖索引
  • 联合索引存在最左匹配原则，遇到范围查询就会停止匹配
  • 索引下推优化：在联合索引遍历过程中，对联合索引包含的字段先做判断，直接过滤掉不满足条件的记录，减少回表次数
    （执行计划中，出现 Extra 为 Using index condition，就说明使用了索引下推）
  • WHERE / GROUP BY / ORDER BY 里没有用到的字段，或者字段中大量重复数据，或者表数据太少，或者字段值经常更新的场景，都不适合建立索引

为什么采用 B+ 树？

B+ 树只在叶子节点存数据，单个节点数据量更小，相同磁盘 IO 能查询更多的节点，另外树高稳定，并且叶子节点双链表的结构也适合范围查找。

索引优化

前缀索引优化：减小索引字段大小，增加一个索引页中存储的索引值，提高索引查询速度。
（order by 就无法使用前缀索引；无法把前缀索引用作覆盖索引）

覆盖索引优化：可以建立联合索引，避免回表查询

主键索引自增：插入新纪录都是追加操作，不需要移动数据，效率会提高；否则可能需要将数据移动到另一个页面（页分裂），还可能造成大量内存碎片

主键字段长度不要太大，主键字段长度越小，意味着二级索引叶子节点越小，占用的空间也就越小

索引最好设置为 NOT NULL：可为 NULL 的列索引统计、值比较都要复杂一些；行格式占用至少一个字节来存储 NULL 值列表

索引失效

  • 模糊匹配（左模糊、左右模糊）
  • 查询条件中对索引列做计算、函数、类型转换
    （v8.0 增加了函数索引，可以针对函数计算后的值建立一个索引）
  • 不按最左优先的方式进行索引匹配
  • WHERE 子句中，OR 前面是索引列，后面不是

执行计划 type 字段表示数据扫描类型：ALL（全表扫描），index（全索引扫描），range（索引范围扫描），ref（非唯一索引扫描），eq_ref（唯一索引扫描），const（结果只有一条的主键或者唯一索引扫描）

执行计划 extra 字段：filesort（查询语句包含分组操作，并且无法利用索引完成排序），temporary（临时表保存），index（覆盖索引，避免回表）

分页

limit offset, size 中，offset 值越大，sql 执行速度越慢；offset 过大时会引发深度分页问题（此时，查询优化器可能选择全表扫描而不是利用索引优化查询），需要通过限制查询数量或者分批获取的方式进行规避。

事务

A 原子性，要么全部完成，要么全部不完成 undo log
C 一致性，事务执行前后数据满足完整性约束，数据库保持一致性状态 通过 A + I + D 来保证
I 隔离性，防止多个并发事务交叉执行导致数据不一致 MVCC / 锁
D 持久性，事务处理完成后的修改就是永久的 redo log

脏读：一个事务读到了另一个未提交事务（也就是可能回滚）修改过的数据
不可重复读：一个事务多次读取同一个数据，前后不一致
幻读：一个事务多次查询某个符合条件的记录数量，前后不一致

快照读与当前读

快照读（select）：通过 MVCC（多版本并发控制） 解决幻读
当前读（select … for update）：通过 next-key lock（记录锁+间隙锁）解决幻读

四种隔离级别怎么实现

读未提交，直接读取即可；串行化通过加读写锁来避免并行访问。

读已提交和可重复读通过 Read View 来实现，读已提交在每个语句执行前都会重新生成一个 View，而可重复读在启动事务生成一个 View。

记录锁、间隙锁（锁住范围，但不包含记录本身）、临键锁（锁定范围和记录本身，左开右闭）

内存

InnoDB 设计了一个 Buffer Pool（默认 128M） 来提高数据库读写性能

修改数据时，首先修改缓存池中数据所在的页，然后将页设置为脏页，最后由后台线程将脏页落盘

维护 Free 链表管理空闲页，维护 Flush 链表管理脏页，维护 LRU 链表管理脏页和干净页

脏页什么时候落盘？

  • redo log 满了，主动触发
  • 缓存池不足而淘汰脏页
  • MySQL 空闲，后台线程定期刷盘
  • MySQL 正常关闭之前

MySQL 的 LRU 有什么不同？

简单的 LRU 无法避免 预读失效（根据程序空间局部性，MySQL 加载数据页会把相邻数据页一并加载，但是提前加载的数据页并没有被访问）和 缓存池污染（SQL 语句扫描大量数据，缓存空间有限，可能会把缓存页换出，导致热数据淘汰）

MySQL 将 LRU 划分为 young 和 old 区域，预读页加载到 old 区头部，真正被访问时再插入到 young 区域头部

MySQL 对进入 young 区域增加了 old 区域停留时间的判断

MySQL 针对 young 区域还有一个优化，为了防止 young 区域节点频繁移动到头部，规定只有后面的 3/4 被访问了才会移动

日志

更新语句的执行中，会涉及到 undo log（实现原子性），redo log（实现持久性），binlog（主要用于数据备份和主从复制）

一条记录的每一次更新操作产生的 undo log 都有一个 roll_pointer 指针（将 undo log 串成版本链）和一个 trx_id 事务 id（知道记录是被哪个事务修改的）

undo log 怎么刷盘？

undo log 和数据页刷盘是一致的，都需要通过 redo log 进行持久化

缓存池中有 undo 页，对 undo 页的修改也会记录到 redo log

redo log 是什么？redo log 的必要性？

redo log 是物理日志，记录某个数据页做了什么修改。事务提交时，只要先将 redo log 持久化到磁盘即可，可以不需要等到缓存池脏页数据持久化，系统重启可以根据 redo log 恢复数据。

生成的 undo log 会写入 Undo 页面，内存修改此页面，也需要记录对应的 redo log。

需要 redo log 的原因：（1）崩溃恢复（2）随机写 变为 顺序写

binlog？

记录所有数据库表结构变更和表数据修改的日志，不会记录查询操作。

前面 undo log 和 redo log 都是 InnoDB 存储引擎生成的，MySQL 完成更新操作后，Server 层还会生成一条 binlog，事务提交之后，把产生的 binlog 统一写入 binlog 文件。

Problems

Hash 索引与 BTree 索引有什么区别？

数据结构：Hash 基于哈希表，B-Tree 基于树结构。
适用场景：Hash 适合等值查询，B-Tree 相对适合范围查询。
性能：Hash 快速但不支持排序，B-Tree 支持排序。
存储空间：Hash 紧凑，B-Tree 详细。

怎么把一张表的部分数据更新到另一张表？

使用 UPDATE JOIN，UPFATE 确定要更新的表，再用 JOIN 子句将两张表连接起来，根据条件更新目标表的数据。

聚簇索引与非聚簇索引的区别？

聚簇索引与非聚簇索引的区别在于数据存储方式和访问效率。
聚簇索引将数据行存储在叶子节点，数据的物理顺序与索引顺序一致，适合范围查询和排序操作；
而非聚簇索引的叶子节点存储的是数据行的指针，数据的物理顺序与索引顺序无关，适合快速查找特定值。
聚簇索引通常只能有一个，而非聚簇索引可以有多个。

谈谈 SQL 注入及防范

SQL注入是指攻击者插入恶意 SQL 代码攻击数据库，产生数据泄露等风险。
防范措施包括：

  • 参数化查询（将 SQL 代码与输入分离，使得只能输入数字或字符串，而无法输入代码）
  • 输入验证
  • 最小权限原则
  • …

INNER / LEFT / RIGHT JOIN 区别？

INNER JOIN 只取交集，LEFT JOIN 保留左表全部，RIGHT JOIN 保留右表全部，未匹配部分用 NULL 填充，实际应用时根据业务需求选择合适的连接方式。

数据库为什么使用B+树而不使用红黑树？

B+树更适合磁盘存储和范围查询，B+数包含多个键值，减少了树的高度，从而减少了磁盘IO的次数，提高查询效率。B+树的叶子节点通过链表链接，支持于顺序查询和范围查询。红黑树是二叉树，且节点大小固定，存储大量数据时，树高增长过快，查找性能下降。

大表（千万级）查询 / 维护该怎么办？

索引优化，拒绝全表扫描，语句调优 避免 SELECT *，明确指定所需字段，减少I/O开销，分区，分页，分表

如何高效批量插入数据？

  • 使用批量插入语句，一次插入多行
  • 插入前暂时禁用索引，插入后再重建索引
  • 多个插入操作放在一个事务，减少事务提交的开销

DELETE / TRUNCATE / DROP 区别？

delete 删除数据行，drop 删除表数据和表结构，truncate 清空表的所有数据

LIMIT 100000000,10 与 LIMIT 10 性能差异？

LIMIT 100000000, 10 等价于 LIMIT 10 OFFSET 100000000, 都用于跳过指定的 “偏移量” 行, 返回指定条数的行
即使只取 10 行，也需从头遍历到第 1 亿行，过程中会读取大量无关数据，查询速度极慢。
OFFSET 仅适合小偏移量（如前几页）分页，大数据量分页通常需用“游标”(基于主键/唯一索引的范围查询) 替代，如 WHERE id > 100000000 LIMIT 10

WHERE 与 HAVING 的区别？

where 分组前过滤不可以使用聚合函数，having 分组后过滤可以使用聚合函数

联合索引的存储结构与最左前缀原则

联合索引的存储结构是将多个列的值组合在一起形成一个复合键，按照字典序进行排序存储。
最左前缀原则指的是在使用联合索引时，查询条件必须从索引的最左列开始，才能有效利用索引进行加速查询。

如何使用 EXPLAIN 分析查询？关注哪些列？

使用 EXPLAIN 分析查询只需要在SELECT语句前加上“EXPLAIN”即可。
关注的列有：

  1. type访问类型；
  2. rows预估需要扫描的行数；
  3. keys实际使用的索引；
  4. extra额外信息；
  5. possible_keys可能使用的索引；
  6. id执行顺序；
  7. select_type查询类型；
  8. tables查询涉及的表等。

数据库分页的实现方式？

一般是通过 LIMIT 和 OFFSET 语句结合实现分页，LIMIT返回记录的数量，OFFSET指定记录的起始位置，但是数据量较大时，可以使用基于索引的分页方式，就是记录上次查询的最后一个主键值来获取下一页的索引，从而避免使用 OFFSET，不需要读取并跳过之前的数据，提高查询速度。

B+ 树查询数据的全过程？

在B+树中，由叶子节点存储实际的数据行记录，查询数据是层级递归与页内二分+链表遍历的结合
查找过程：

查询从B+树的根节点开始，通过比较目标键值与节点中存储的索引键值（通常采用二分查找），确定数据落在哪个区间，从而选择对应的分支继续向下查找。这一过程逐层递归，从上到下最终定位到目标叶子节点。
叶子节点大小为16KB，存储的数据行不止一条，对叶子节点中数据行以组的形式划分，利用页目录结构，通过二分查找页目录中的槽，可以快速定位到包含目标数据行的组。
定位到目标组后，通过链表遍历（通常是单向或双向链表）逐条检查数据行，直到找到完全匹配的记录。

索引失效后怎么办？

  1. 优化sql结构，满足最左前缀原则；
  2. 尽量减少索引列的数学计算；
  3. 字段类型尽量符合定义，减少隐式转换
  4. 根据需要优化索引结构，提高索引的选择性

EXISTS 与 IN 的区别？

EXISTS 遍历外表每一行，检查下内表有没有存在，适合外表小内表大的场景

IN先执行内表查询，再把结果集进行匹配，适合内表小外表大的情景

坑点

mysql 四个隔离级别的主要区别就在于能否在 transaction 中读取到其他 transaction 正在改动的数据

隔离级别 解释
Serializable 不受任何影响，只能看到 transaction 开始的全局景象
Repeatable Read 允许看到别人 insert 或者 delete 后的结果
Read committed 在此基础上，允许看到别人 update 后的结果
Read uncommitted 其他 transaction 还没有保存的修改都可以看到

mysql 奇怪的设定：如果命令里只有 select 查询，执行 Repeatable Read；如果混入 insert 命令，那么 select 就执行 Read committed。

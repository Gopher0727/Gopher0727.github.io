基础

网络体系结构的目的是解决主机之间网络通信问题。

  • OSI
    • 物理层、数据链路层、网络层、传输层、会话层、表示层、应用层
  • TCP/IP
    • 网络接口层、网络层、传输层、应用层
  • 五层协议
    • 物理层、数据链路层、网络层、传输层、应用层

物理层：考虑使用何种信号来传输比特

ARP / ICMP / IGMP

数据链路层：分组在一个网络（一段链路）上的传输

网络层：分组在多个网络传输（路由）

传输层：端到端，进程之间基于网络的通信问题

应用层：通过应用进程的交互来实现特定的网络应用

基于 TCP：

  • FTP
  • SMTP / POP3
  • HTTP

基于 UDP：

  • DNS
  • DHCP

问题

CDN？

CDN，内容分发网络，本质是一套‘分布式的内容缓存与分发系统’。
它通过在全球/全国部署大量‘边缘节点’，将源站的静态内容（如图片、视频、JS/CSS 文件）缓存到离用户最近的节点，核心解决两个问题： 1. 降低用户访问延迟； 2. 减轻源站压力（避免所有请求都涌向源站，尤其高并发场景）

应用层报文如何传输到另一台主机？

应用层报文要通过 TCP/IP 四层逐层封装才能传过去：

  1. 应用层生成报文（比如 HTTP 请求）； ​
  2. 传输层加 TCP 头部（源/目的端口、序列号，管端到端），变成 TCP 段； ​
  3. 网络层加 IP 头部（源/目的 IP、TTL，管跨网段路由），变成 IP 包； ​
  4. 网络接口层加以太网帧头帧尾（源/目的 MAC，管硬件传输），变成帧，通过 WiFi/网线发出去； ​
  5. 中间路由器按 IP 选路转发，到目标主机后，从下到上逐层拆包（去帧头→去 IP 头→去 TCP 头），最后把应用层报文交给对方应用（比如百度的 HTTP 服务）。

直播业务通常使用什么协议？

直播协议按场景选，核心分三类：

  1. RTMP：基于 TCP，延迟1-3秒，是主播推流到服务器的标配，适合传统秀场、游戏直播； ​
  2. HLS/DASH：基于 HTTP，延迟3-10秒，兼容性极强（所有设备支持），适合世界杯这种大规模、对延迟不敏感的直播； ​
  3. WebRTC：基于 UDP，延迟100ms-500ms，接近实时，适合连麦、PK这种互动直播，能满足双向实时通信。

ARP 和 RARP 分别是什么？有什么区别？

ARP 和 RARP 都是解决 IP 与 MAC 映射的协议，但方向相反：
ARP 是从 IP 找 MAC，比如你知道隔壁电脑 IP，发数据前用 ARP 广播问它的 MAC，是局域网通信的基础；
RARP 是从 MAC 找 IP，早期无盘机启动时只有 MAC，靠 RARP 向服务器申请 IP 才能联网。
区别核心是：ARP 正向解析、所有设备用、不用服务器；RARP 反向解析、无盘机用、必须靠服务器。
现在 RARP 基本被 DHCP 替代了，但 ARP 还是网络通信的刚需。

DNS

DNS就是域名系统，帮助用户用域名转化成ip去访问服务器
解析过程：先到浏览器缓存中找，找不到去本地缓存，再去递归DNS服务器，递归DNS就是替用户完成后续查询，用户只需等待结果。没找到再去根DNS服务器，顶级域DNS服务器，权威DNS服务器。递归DNS从权威DNS中拿到ip后，会将结果缓存，按照权威DNS返回的TTL时间决定缓存时长。
其实90%以上的DNS都无需走到权威DNS就能通过缓存命中获取ip，这极大减少全球DNS服务器的请求量，降低用户延迟。

DNS劫持了解吗？如何应对？

  1. 修改客户端缓存
  2. 在 DNS 任意链路劫持响应并修改

防御措施 1. 端到端签名验证（DNSSEC） 2. 传输加密

IPv4 和 IPv6 有什么区别？

IPv4 和 IPv6 的核心区别体现在四个方面：
一是地址空间，IPv4 32 位约 43 亿地址（已耗尽），IPv6 128 位地址无限，解决物联网需求；
二是头部结构，IPv6 精简为固定 40 字节，删除校验和、分片字段，转发效率更高；
三是安全性，IPv6 原生支持 IPsec，数据加密和身份认证；
四是功能，IPv6 取消广播用多播/任播，支持自动配置，无需 NAT，端到端通信更简单

DHCP？

没有 DHCP 的网络会面临两大核心问题，这也是它存在的根本原因：
手动配置效率极低：如果一个网络有 100 台设备，每台都需要手动设置 IP 地址、子网掩码、网关等参数，不仅耗时，还容易出现重复 IP（导致设备冲突断网）、参数输错等问题。
IP 地址资源浪费：设备不联网时，手动分配的 IP 会一直被占用，无法给其他设备使用。DHCP 能动态回收闲置 IP，让有限的 IP 地址资源循环利用。

DHCP 的工作过程可以概括为 “四步握手”，全程由 DHCP 服务器主动分配，客户端无需手动操作，具体流程如下：
第一步：客户端请求 IP（Discover）
新设备接入网络后，会发送一个 “DHCP 发现” 广播包，询问网络中是否有 DHCP 服务器能分配 IP。此时客户端还没有 IP，广播包会发送给网络内所有设备。
第二步：服务器响应（Offer）
DHCP 服务器收到请求后，会从自己的 “IP 地址池” 中挑选一个未被占用的 IP，再发送 “DHCP 提供” 包给客户端，告知其可使用的 IP 及子网掩码、网关等参数。
第三步：客户端确认选择（Request）
若客户端收到多个服务器的 “Offer”（比如网络中有多台 DHCP 服务器），会选择其中一个，然后发送 “DHCP 请求” 广播包，告知所有服务器 “我已选择某台服务器的 IP”，未被选中的服务器会收回自己提供的 IP。
第四步：服务器最终确认（Acknowledge）
被选中的 DHCP 服务器收到 “Request” 后，会发送 “DHCP 确认” 包，正式将 IP 地址分配给客户端

NAT？

NAT的核心作用就两点：

  1. 解决IPv4地址耗尽问题 —— 让内网多设备（手机、电脑）共享一个公网IP上网，通过‘私有IP+内网端口→公网IP+公网端口’的映射，把重复使用的私有IP转换成唯一公网IP，比如家用路由器就是靠NAT让全家设备联网
  2. 提升内网安全性 —— 外网只能看到NAT设备的公网IP，看不到内网设备的私有IP（私有IP公网不可路由），相当于隔离了内网，减少外网直接攻击风险。

内网与外网通信的过程？

内网与外网通信的核心是‘NAT转换+路由转发’，因为内网用私有IP（公网不可路由），必须通过路由器的NAT换成公网IP才能互通

  1. 内网电脑发请求，封装数据包； ​
  2. 路由器收到后，做NAT转换：把源IP换成自己的公网IP，同时记录映射表（内网IP+端口 ↔ 公网IP+端口），方便后续回传； ​
  3. 转换后的数据包通过公网路由到服务器，返回响应（目的IP是路由器公网IP，目的端口是映射的公网端口）； ​
  4. 路由器查映射表，把响应的目的IP换回内网私有IP，转发给电脑； ​
  5. 电脑接收响应，完成通信。
    关键是NAT解决了私有IP不能公网路由的问题，映射表保证响应能准确回传，默认只能内网主动访问外网，要外网访问内网需配端口映射。

Linux 系统是如何收发网络包的？

发: 程序调用send() -> 用户数据拷贝到到socket发送缓冲区 -> 协议栈包装 -> 驱动(用于通知网卡取数据) -> DMA读取数据到网卡 -> 网卡 -> 发送光信号/电信号
收: 网卡数据到达 -> 通过DMA将数据写入环形缓冲区并触发硬件中断 -> 驱动处理数据 -> 协议栈解析 -> 放到socket内核缓冲区 -> 程序read()

IP 协议包含哪些字段？

IPv4 头部核心字段集中在固定20字节里，关键的有： 版本（IPv4=4）、 头部长度（标识头部结束位置）、 总长度（整个包大小）、 标识+标志+片偏移（用于分片重组）、 TTL（防路由环路，每跳减1）、 协议（指明上层是TCP/UDP/ICMP）、 源/目的IP（路由的核心）、 校验和（验头部完整性）。

路由协议？

分三部分：本地查找、路由跳跃、目的网络内传递

  1. 先使用子网掩码计算网络号，如果和自己在一个网络->ARP，否则转发给路由器
  2. 路由器会做三个事情
    2.1 根据目的ip和路由表中的网段进行匹配
    2.2 选择网段中匹配长度最长的那个路由器的ip
    2.3 用ARP找目标路由器的mac，重新封装数据包发送
  3. 目的路由器发现网段就是自己这个网段，ARP找mac

如何获取客户端真实 IP？

  1. 如果是直连 -> getRemoteAddr
  2. 如果是代理 -> getHeader(“x-Forwarded-For”)获取调用链,注意使用累加调用
  3. 确保调用链可信

GET 与 POST 区别？

  1. 从概念上：GET是从服务器获取数据，不会修改数据；POST则是提交数据给服务器进行处理；
  2. 从安全上：GET将数据直接拼接在URL请求后面，安全性低，而POST请求是指数据放在请求体中，不会暴露在URL地址中，安全性更高；
  3. 从幂等性上：GET是幂等的，多次提交的结果是相同的；POST是非幂等的，多次提交可能产生不同的效果

HTTP vs. HTTPS 有什么区别?

  1. HTTP是明文传输，HTTPS加了TLS协议，是加密传输，更安全；
  2. HTTP只需要TCP三次握手过程，而HTTPS还增加了TLS握手过程；
  3. HTTP端口号是80，HTTPS端口号是443；
  4. HTTPS需要通过CA（证书权威机构）申请数字证书，确保服务器是可信的；

HTTP 请求头中到底包含什么？

  1. 请求行：请求方法（GET/POST）+请求目标（URL地址）+协议版本；
  2. 首部字段（Header）：以键值对的形式存储，一行一个键值对，包括Host（服务器域名），User-Agent（客户端身份，比如浏览器型号），Accept（客户端能接受的数据格式），Cookie（客户端存储的身份信息，唯一的ID），Content-Type（客户端告知服务器请求体的数据格式）；
  3. 空行：表示请求头结束；
  4. 请求体：包含JSON格式的数据。

SSE（Server-Sent Events）与 WebSocket 有什么区别?

SSE 是服务器单向推，客户端发数据还得用 HTTP； WebSocket 是双向全双工，双方能主动发。

底层上，SSE 基于 HTTP 长连接，不用升级协议，只支持文本，浏览器原生自动重连，适合新闻推送、日志监控这些单向场景；WebSocket 要 HTTP 握手升级协议，支持文本和二进制，需手动实现重连，适合聊天、弹幕这些双向交互场景。

HTTPS 的「秘钥交换 + 证书校验」全流程

①、客户端向服务器发起请求。
②、服务器接收到请求后，返回自己的数字证书，包含了公钥、颁发机构等信息。
③、客户端收到服务器的证书后，验证证书的合法性，如果合法生成一个会话密钥，然后用服务器的公钥加密成随机码，发送给服务器。
④、服务器收到被公钥加密的随机码，用私钥解密，得到会话密钥。
⑤、客户端和服务器通过会话密钥对通信内容进行加密，然后传输。

如果通信内容被截取，但由于没有会话密钥，所以无法解密。当通信结束后，连接会被关闭，会话密钥也会被销毁，下次通信会重新生成一个会话密钥。
